

## 1. 为什么要统一规范？

-   **快速定位**：通过提交信息一眼看出是新增功能、修复 Bug 还是维护杂务。
-   **自动化生成 Changelog**：CI/CD 工具可根据提交类型自动生成版本发布日志。
-   **提升 Code Review 效率**：Reviewer 无需深入代码即可了解变更意图。
-   **便于回溯**：`git log` 输出清晰，方便查找历史变更原因。

## 2. 提交信息格式

每条提交信息由 **Header**、**Body**（可选）、**Footer**（可选）组成，其中 **Header** 为必填。

```
<type>(<scope>): <subject>
<BLANK LINE>
<body>
<BLANK LINE>
<footer>
```


### 2.1 Header（头部）

#### Type（必填）
用于描述本次提交的类型，**必须**为以下之一：

---

| 类型 | 说明 | 典型例子 |
| :--- | :--- | :--- |
| **`feat`** | 新增一个用户功能 | `feat: 增加微信快捷登录` |
| **`fix`** | 修复一个 Bug | `fix: 修复分页组件页码高亮错误` |
| **`docs`** | 仅文档变更（如 README、Javadoc） | `docs: 更新部署文档中的环境变量说明` |
| **`style`** | 代码格式调整（空格、缩进、分号等），**不影响逻辑** | `style: 统一缩进为2个空格，去除多余分号` |
| **`refactor`** | 代码重构（既不是新功能也不是修 Bug） | `refactor: 抽取订单金额计算为独立工具类` |
| **`perf`** | 性能优化（提升性能的代码变更） | `perf: 优化大图列表渲染，使用虚拟滚动` |
| **`test`** | 增加或修改测试代码（单元测试、集成测试） | `test: 为支付回调接口补充边界条件测试` |
| **`build`** | 影响构建系统或外部依赖（如 webpack、npm、Gradle） | `build: 升级 Vite 从 v4 到 v5` |
| **`ci`** | 修改 CI 配置文件或脚本（如 GitHub Actions、Jenkins） | `ci: 增加代码覆盖率报告上传步骤` |
| **`chore`** | 其他杂务（修改 .gitignore、删除废弃文件等） | `chore: 将 node_modules 加入 .gitignore` |
| **`revert`** | 回退之前的提交 | `revert: 回退 feat: 增加微信快捷登录` |

---

#### Scope（可选）
用括号注明本次修改影响的**模块/范围**，例如：
-   `feat(auth):` 表示修改了“认证模块”
-   `fix(cart):` 表示修复了“购物车模块”
-   `style(ui):` 表示修改了“UI样式”

如果影响范围较大或不明确，可以省略 Scope。

#### Subject（必填）
用一句话简明扼要描述本次提交做了什么，**不超过 50 个字符**。

-   使用**中文**描述（团队内部统一便于阅读）。
-   使用**祈使句**，现在时态（如“添加”而不是“添加了”）。
-   结尾**不加句号**。

**示例**：
```
feat(user): 增加手机号快捷登录
fix(pay): 修复支付宝回调验签失败问题
```



### 2.2 Body（正文，可选）

当提交内容较复杂时，在 Body 中详细解释 **“为什么做”** 以及 **“如何实现的”**。

-   每行不超过 72 个字符。
-   解释变更的动机，以及与之前实现方式的差异。

**示例**：
```
fix(order): 修复重复扣减库存并发问题

由于高并发场景下使用乐观锁重试机制不当，导致同一订单多次扣减。
改用分布式锁（Redis + Redisson）保证原子性，并增加重试失败后的回滚日志。
```

---

### 2.3 Footer（页脚，可选）

主要用于两种情况：

#### （1）关联 Issue 或需求单号
```
feat(report): 新增日活报表导出功能

Closes #JIRA-1234
```

#### （2）引入 Breaking Change（不兼容变更）
如果本次提交会破坏现有 API 或接口，需要在 Footer 中以 `BREAKING CHANGE:` 开头说明。

```
feat(api): 重构用户信息接口

将 /api/v1/user 的返回字段名由 userName 改为 username。
BREAKING CHANGE: 旧版客户端需同步更新字段名。
```

## 3. 提交信息示例（完整）

### 简单提交（仅 Header）
```
docs(readme): 更新环境搭建步骤
```

### 带 Body 和 Footer 的提交
```
feat(payment): 接入微信支付 V3 接口

替换旧版 V2 接口，支持商家分账功能。
新增回调通知处理模块，增强异常重试机制。

Closes #PAY-888
```

---

## 4. 实战建议
 **频繁提交，但保持原子性**：每次提交只做一件事（如一个 Bug 修复或一个功能点），不要混杂多个不相关的改动。
